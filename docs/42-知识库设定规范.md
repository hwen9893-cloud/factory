# 42 - 知识库（center.knowledge）设定规范

> Version：v1.0　Last Update：2026-06
> Status：Active
> 依赖：`11-模块命名规范.md`、`41-修仙爽文角色量化系统.md`、`61-数据契约与Schema.md`、`52-一致性技术预研方案.md`

---

# 1. 文档目的

定义 `center.knowledge` 的实体类型、ID 规范、录入流程与冲突检查规则，作为剧情设计的工作底座，并为 `center.qa` 的设定一致性校验提供事实源。

---

# 2. 知识实体类型

| 实体 | KnowledgeID 前缀 | 说明 |
|------|-----------------|------|
| 角色 | `knowledge.character.*` | 与 `41` 角色卡一一对应（含七维分、关系） |
| 宗门/势力 | `knowledge.sect.*` | 势力设定、层级、立场 |
| 法宝/功法 | `knowledge.artifact.*` | 法宝/功法的能力、限制、归属 |
| 地图/地点 | `knowledge.place.*` | 地理、势力范围 |
| 境界体系 | `knowledge.realm.*` | 战力坐标体系（对接 `41` G3） |
| 时间线 | `knowledge.timeline.*` | 事件先后、因果 |
| 关系 | `knowledge.relation.*` | 角色间关系（对接 `41` F 层） |

> ID 三段式 `{module}.{entity}.{id}` 以 `11` §2.2 为准，落地 schema 见 `61`。

---

# 3. 设定录入流程

```
人工起草（剧情设计）
   │  用 `41` 的角色卡模板 / 设定模板
   ▼
LLM 辅助补全（skill.knowledge.draft，可选）
   │  本地模型扩写，但不得编造硬设定
   ▼
人工确认 → 写入 center.knowledge（带 lineage）
   │
   ▼
触发 center.qa 增量校验（与既有事实 diff）
```

- 录入即分配 KnowledgeID，后续所有引用只用 ID（呼应 Asset/Knowledge First）。
- 硬设定（境界体系、关键时间线）改动需走"重评"流程（见 `41` §八）。

---

# 4. 设定冲突检查规则集（喂给 center.qa）

| 检查项 | 规则 | 冲突示例 |
|--------|------|---------|
| 人物属性一致 | 角色发色/瞳色/境界等不得前后矛盾 | 第 5 章金丹，第 3 章已元婴 |
| 时间线单调 | 事件因果不得倒置 | 师父死后又出场（非诈尸设定） |
| 境界坐标合法 | 战力不得超出体系定义 | 练气期越级杀化神（无合理金手指） |
| 关系自洽 | 关系类型不得自相矛盾 | 同时为"生死宿敌"与"道侣"无转化 |
| 命名唯一 | 同一实体不得多 ID | 一个角色两个 character ID |

> 规则集随连载迭代扩充；每条规则对应 `72-评测工程EvalHarness.md` 中的一个测试用例。

---

# 5. 世界观一致性基线（Phase 1 验收）

对应 `22-成功标准与里程碑.md` 标准 #1：

- `center.knowledge` 覆盖 **≥ 30 个角色**、**≥ 200 条设定条目**。
- 新章节自动触发 `center.qa` 检查，对人工注入的设定冲突测试集**召回率 ≥ 80%**。

---

# 6. 与 factory.novel 的契约

```
factory.novel 的每个 skill：
  1. 先 *查询* center.knowledge（取相关角色/设定/时间线）
  2. 再生成（不得脱离查询结果自由发挥硬设定）
  3. 产物经 center.qa 校验
  4. 新增设定 *写回* center.knowledge（带 lineage）
```

铁律（呼应 `31` §4）：factory **不直接生成知识**，必须查询 → 生成 → 写回，保证可追踪、可回滚。

---

# 7. 长连载记忆管理（≥ 50 章后的关键问题）

## 7.1 问题：知识库无界增长 + 上下文窗口瓶颈

连载至第 50 章时，`center.knowledge` 已有数百条设定条目，`knowledge.timeline` 记录了近千个事件。**直接全量注入上下文窗口不可行**（超出 32K 限制）。

## 7.2 分层检索策略（代替全量注入）

```
skill.novel.chapter 生成前的知识查询流程：

① 精确查询（必注入，约 600 token）
   → 当前章涉及的角色 ID → 查 character payload
   → 当前场景地点 ID → 查 place payload

② 语义检索（相关设定，约 800 token）
   → 当前章大纲 + 上章结尾 → embedding 检索 center.knowledge
   → top-K（K=5–8）最相关设定条目注入

③ 时间线摘要（约 400 token）
   → 查 knowledge.timeline，仅取"当前章前 5 个事件 + 当前卷关键事件"
   → 跨卷用"卷摘要"代替全量事件列表

总注入 ≈ 1800 token，在 34 §5 的上下文预算范围内。
```

## 7.3 卷快照（Volume Snapshot）机制

每完成一卷时，自动生成"卷摘要快照"写入 `center.knowledge`：

```json
{
  "knowledge_id": "knowledge.snapshot.vol001",
  "entity": "volume_snapshot",
  "payload": {
    "volume_no": 1,
    "ch_range": "1–30",
    "char_state_summary": {
      "knowledge.character.0001": {
        "realm_at_end": "金丹期·后期",
        "key_events": ["拜师失败→自立门户", "获得噬魂剑"],
        "relations_changed": [{"to": "knowledge.character.0002", "new_rel": "宿敌"}]
      }
    },
    "major_events": ["宗门大比", "秘境开启", "BOSS 初战（败）"],
    "world_changes": ["血月宗崛起", "炼器谷关闭"],
    "unresolved_threads": ["天机老人的预言", "血脉秘密"]
  }
}
```

第 N+1 卷起，只需注入 Vol 1–N 的快照（每卷约 200 token），而非完整事件流。

## 7.4 设定演进规则（角色境界/关系变化）

| 演进类型 | 处理方式 |
|---------|---------|
| 境界升级 | 在 character.payload 的 `realm` 字段追加新值；旧值保留在历史版本（lineage 回滚可查） |
| 关系反转（宿敌→盟友） | `center_knowledge_relation` 不覆盖，新增一条同 from/to 的记录，带 `from_chapter` 字段 |
| 角色死亡 | 将 `status` 改为 `dead`，死亡章节写入 `knowledge.timeline`；死角色依然可被 QA 引用（避免复活 bug） |
| 硬设定变更（如境界体系重构） | 需走"重评"流程：剧情设计拍板 → 更新 `knowledge.realm` → 触发批量 QA 全库回归 |

> **铁律**：设定演进只追加记录，不删除历史状态。任何历史章节的上下文都可根据时间线快照完整重建。
