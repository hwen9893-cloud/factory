# 1. 文档目的

`52-一致性技术预研方案.md` 定义了一致性的四个维度与 M3.5 验收门（"做什么 / 验什么"）。本 Playbook 回答"**在 Mac Studio + 云 GPU 的现实硬件下，每个维度具体怎么落地、哪些本地做、哪些上云**"（"怎么做"）。

> 核心前提（见 `32` §3）：**LoRA 训练与视频生成是 CUDA 生态，必须云 GPU**；Mac 负责 spec 准备、轻量推理试色、设定 QA。

---

# 2. 四维度落地路线（标注 Mac / 云）

| 维度 | 手段 | 执行位置 |
|------|------|----------|
| **身份一致（图像）** | 每角色 LoRA + canonical reference sheet（多角度定妆图） | **LoRA 训练：云**；reference 生成/试色：Mac 可小批量 |
| | IP-Adapter / InstantID 类 reference-based 身份锚定 | 推理可 Mac 试 / 放量上云 |
| | 固定 seed 区间 + ControlNet 控姿态 | 云（ControlNet 全套在 CUDA 镜像） |
| | 兜底：face restoration / 局部重绘 | 云为主 |
| **跨模态（文→图）** | appearance spec 结构化注入（发色/瞳色/服饰/标志物/体型） | **Mac（确定性注入，无需 GPU）** |
| | Prompt 由 spec 确定性生成，禁止 LLM 每次重述 | Mac（本地 LLM 组 prompt） |
| **时序（视频）** | 关键帧锁定 → reference 生成中间帧 | **云** |
| | 末帧→首帧链式衔接长镜头 | 云 |
| | 带 reference 条件的视频模型 | 云 |
| | 兜底：逐帧 face swap / restoration | 云 |
| **设定（QA）** | QA Agent 将新章节/分镜与 `center.knowledge` diff | **Mac（本地 LLM + 规则）** |

---

# 3. appearance spec 字段（跨模态核心，Mac 本地）

角色外观规格结构化存入 `center.asset`，由 spec **确定性注入** prompt（字段落地见 `61`）：

| 字段 | 例 |
|------|-----|
| 发色/发型 | 玄色长发束冠 |
| 瞳色 | 金瞳（异瞳设定） |
| 服饰 | 青色道袍 + 玄铁护腕 |
| 标志物 | 左眉断痕 / 古剑「噬魂」 |
| 体型/年龄感 | 清瘦少年 |
| 气质标签 | 阴冷、不羁 |

> 这是避免"LLM 措辞漂移→视觉漂移"的关键，且**完全在 Mac 本地完成**，不消耗云 GPU。

# 5. M3.5 验收门（工具链落到 Mac/云）

四维度基线见 `52` §4。本 Playbook 补充执行位置：

| 维度 | 测试 | 执行 |
|------|------|------|
| 身份一致 | 主角生成 N=20 张，ArcFace 余弦相似度 ≥ 0.6，人工判定一致率 ≥ 90% | 生成上云，评测脚本 Mac/云均可（见 `72`） |
| 跨模态 | 描写生成图，K 个外观属性命中率 ≥ 85% | spec 注入 Mac，生成上云 |
| 时序 | 1 段 10 秒片段，帧间漂移低于阈值，人工通过率 ≥ 80% | 云 |
| 设定 | 注入冲突测试集，QA 召回率 ≥ 80% | Mac |

---

# 6. de-scope 预案

当某维度达不到基线（尤其时序/视频，Mac+云成本高时）：

- 单维度未达标 → 在该维度迭代方案或更换技术路线，**不放行**。
- 多维度系统性未达标 → 触发 de-scope：**先做半动态分镜 / 动态漫，而非全视频**，重排 M4 范围（呼应 `21` 的有声书/动态漫优先策略）。
