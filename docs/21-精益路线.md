> Status：Active

# 0. 核心判断（先读这一段）

原规划的隐含逻辑是 **"建平台 → 产内容"**：先把 `infra` / `center.knowledge` / 编排 / 资产中心全部搭好，再产出内容。这个顺序对小团队是反的。

修正后的逻辑是 **"先手工产内容 → 找到跑得通的形式 → 只为那一个形式做自动化"**。

> 3 人版的调整：相比 1–2 人，剧情与美工可以**并行推进**（剧情写设定、美工试立绘风格），代码负责把验证过的形式自动化。但**先验证后建厂**的内核不变。
>
> 硬件附带结论：本项目最硬的技术赌注是**全动画视频的一致性**（见 `52-一致性技术预研方案.md`）。而 **Mac Studio 恰好最不擅长视频扩散与 LoRA 训练**（CUDA 生态），这从硬件层面就把全动画自然推后。若市场测试证明**有声书 / 动态漫**就能跑通，则 v1 直接绕开该未解难题。**Phase 0 的结果决定要不要现在啃一致性。**

# 1. Market-First 测试片（Phase 0）

## 1.1 目标

用 ≤ 4–6 周、最小花费，在**不建任何自定义平台**的前提下，回答四个问题：
**目标受众是谁 / 哪个平台有反应 / 哪种形式先跑通 / 有没有变现路径。**

## 1.2 设计原则

| 原则 | 做法 |
|------|------|
| 零平台 | 全用现成工具：LLM 辅助写作 + TTS（Mac 本地可跑）+ 剪辑软件（CapCut 等），不写一行平台代码 |
| 单一变量 | 选**一个**修仙故事种子（强钩子开篇），跨形式复用，隔离"形式"这个变量 |
| 成本优先出信号 | 只做两种最便宜、最快出信号的形式，**不做全动画**（最贵、一致性最难、Mac 也跑不动） |

## 1.3 两个候选形式（按成本由低到高）

| 形式 | 工具链 | 一致性风险 | Mac Studio 适配 | 候选发布平台 |
|------|--------|-----------|----------------|-------------|
| **A. 有声书 / 广播剧片段** | 文本 → 多角色 TTS → 简单 BGM | **零视觉一致性问题** | ✅ 本地 TTS 顺手 | 喜马拉雅 / 番茄畅听 / Spotify / YouTube |
| **B. 动态漫 / AI 短视频** | 立绘 + 简单运镜口型 + 配音 | 低（动态漫对静帧容忍度高） | ⚠️ 立绘可本地试，LoRA/视频需云 GPU | 抖音 / 快手 / B站 / YouTube Shorts |

> 全动画刻意推迟：它同时是最贵、最难一致、Mac 最不擅长的，拿它做第一次信号验证是最差选择。
> **建议第一条形式从有声书起步**——它与 Mac Studio 的本地能力最契合，且零视觉一致性风险。

## 1.4 度量与决策线

测试不靠感觉，靠数据。核心指标：**完播率、追更意愿（是否催更/关注）、互动率（评论点赞转发）、涨粉转化、评论情绪**。

> ⚠️ 不设绝对数字阈值——各平台基线差异极大。
> **统一对标"该平台同品类（修仙/AI 内容）账号"的中位水平**，分绿/黄/红：
> - 🟢 绿：完播 + 追更 + 情绪均达到或超过同品类中位 → 该形式可投自动化
> - 🟡 黄：单项达标 → 调内容/钩子再测一轮
> - 🔴 红：系统性低于同品类 → 不是技术问题，是内容/形式问题，**pivot 内容而非加码建平台**

（量化口径与采集方法见 `72-评测工程EvalHarness.md` §2.1。）

## 1.5 变现信号探针

发布同时主动探这几条路是否走得通：平台创作激励/分成、付费收听或付费章节（番茄/喜马拉雅）、短剧分账、IP/版权授权兴趣。

## 1.6 Phase 0 交付物

一页《形式决策》：**选定 形式 × 平台 投自动化**，或判定"内容 pivot"。后续所有工程投入都以这页为依据。

# 2. 团队 / 预算约束（3 人显式写入）

## 2.1 资源约束

> **团队规模：3 人 —— 剧情设计 / 美工设计 / 代码实现。**

3 人且角色互补，比 1–2 人有更强的并行能力，但仍远小于原蓝图（8 模块 / 24 月）所隐含的创业公司体量。**承认这一点，是做正确取舍的前提。** 预算细节见 `23-预算与资源约束.md`。

## 2.2 三人 owning 与并行策略

| 人 | Phase 0 任务 | Phase 1 owning |
|----|-------------|----------------|
| 剧情设计 | 写故事种子、设定、角色卡（用 `41` 量化系统） | `factory.novel` + `center.knowledge` |
| 美工设计 | 试立绘风格、定妆图（仅当形式 B 候选时） | `factory.visual` + `center.asset` |
| 代码实现 | Phase 0 用现成工具，不写平台；准备 `infra` 骨架 | `infra` + `orchestrator` + `center.*` + skills |

> 并行红线：**代码不得在 Phase 0 信号确认前提前建平台**；其精力应放在评测脚本（`72`）与硬件/模型预研（`32`/`33`）。

## 2.3 范围重排（替代原 M1–M7 平台优先顺序）

| 阶段 | 时间 | 范围 | 关卡 |
|------|------|------|------|
| **Phase 0** | 0–6 周 | 市场验证：现成工具产 ≥ 2–3 条内容，测量，决策 | 🟢出信号 / 🔴pivot |
| **Phase 1** | 信号确认后 | **只自动化跑通的那一个形式**：最小流水线 + `center.knowledge` | 人工率可下降 |
| **Phase 2** | — | 规模化已验证形式：多集产出、降人工率、稳定性（含 M3.5 关卡） | 可持续产出 |
| **Phase 3+** | 愿景，不承诺时间 | 扩展相邻形式 / 全动画 / 运营平台 | — |

**与里程碑的对应**：`22-成功标准与里程碑.md` 的 M1–M7 + M3.5 是 Phase 1+ 的内部里程碑，按 Phase 0 选定形式裁剪——只建该形式需要的那几个模块。

## 2.4 关键削减项（3 人 + Mac Studio 视角）

| 削减项 | 处理 | 理由 |
|--------|------|------|
| 自购 CUDA 工作站 | **Phase 1 形式确定前不买**，视觉/视频用云 GPU 突发 | 未验证产品前的过早 Capex；Mac Studio 已是主平台（见 `32`） |
| 全动画流水线 | 推迟到某形式跑通且需要视频时 | 最贵、一致性最难、Mac 跑不动 |
| 一致性预研 / M3.5 | **仅当选定形式含视频时才上关键路径** | 有声书胜出则 v1 无需 |
| Neo4j 图数据库 | 第一阶段不上，PostgreSQL 关系表足够 | 关系复杂度未到（见 `61`） |
| LangGraph 全节点编排 | Phase 0/1 用脚本/半手工 skill，量大了再上完整编排 | 编排为规模服务，规模未到 |
| 运营平台 `center.operation` | 先用各平台原生后台看数据 | 自建分析是规模化后的事 |
| 多世界观 / 多项目通用性 | 先为**一个 IP** 写死 | 通用性是过早抽象 |

# 3. 一句话总结

> 3 人的正确打法不是"造一座完美的工厂"，而是**先用最便宜的手工方式发出去一条内容（优先有声书，契合 Mac）、拿到真实信号、只为信号最强的那条路径建最小自动化**。剧情与美工并行验证内容，代码守住"不提前建厂"的红线。先活下来证明有人看，再谈把它变成工厂。
