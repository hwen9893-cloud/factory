# 3. 交接数据格式（流水线交接点）

```
剧情 ──[角色卡+设定, 写入 center.knowledge]──► 美工
      格式：`41` 的角色卡模板 + `61` §5 的 appearance spec 字段

美工 ──[立绘/定妆+AssetID, 写入 center.asset]──► 代码/动画
      格式：`61` §3.1 的 center_asset_item + appearance_spec(jsonb)

代码 ──[skill/流水线 + lineage]──► 全员
      格式：`61` §4.1 的 lineage_edge（每产物可溯源）
```

> 铁律：交接只通过 **ID + 结构化字段**，不靠口头/聊天记录传递硬设定。

# 4. 评审节奏

| 评审 | 频率 | 参与 | 依据 |
|------|------|------|------|
| 章节评审 | 每章/每卷 | 剧情主审 | `22` 标准 #2、`41` 角色量化 |

# 5. 人在环触发点（必须人工放行）

对接 `62` §5 的 interrupt 节点：

| 节点 | 放行人 | 不放行后果 |
|------|--------|-----------|
| 大纲/设定评审 | 剧情 | 回退重写 |
| 立绘/定妆评审 | 美工 | 回退重生成 |
| M3.5 一致性关卡 | 美工+代码 | 不进动画，触发 de-scope（`51` §6） |
| 发布前终审 | 全员 | 不推送 |

# 6. 冲突解决机制

| 冲突类型 | 处理 |
|---------|------|
| 数值与直觉冲突（角色） | **信直觉，用框架反查原因**（`41` 结尾原则）；剧情拍板 |
| 一致性达不到 | 美工主导技术迭代；多维度不达标 → 全员决策 de-scope |
| 模型/质量分歧 | 以 `72` 盲测数据为准，代码执行，剧情盲评 |
| 范围/优先级分歧 | 回到 `21` Phase 原则：先验证后建厂，市场信号优先 |

> 原则：**用数据和文档解决分歧，而非靠职位或音量。** 所有决策落回对应文档并更新版本。
