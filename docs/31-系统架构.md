# 1. 文档目的

定义 AI Cultivation Studio 的总体软件架构，包括系统组成、模块划分、数据流、Agent/Skill 协作、服务部署与开发规范。

设计准则：**高内聚、低耦合、模块化、可扩展、分层 Local First。**

---

# 2. 总体设计目标

平台需具备：分层本地/云协同部署、长时间稳定运行、Skill 自动协同、可扩展、多模型支持、多项目/多世界观/多语言支持。

# 3. 总体架构（六层）

```
                                  User
                                    │
   ┌────────────────────────────────────────────────────────────┐

                                    │
   ┌────────────────────────────────────────────────────────────┐
   │  L2 Workflow — orchestrator (LangGraph)                     │
   │  节点 = skill 调用：Idea→Outline→Novel→Review→Storyboard     │

   └──────────────┬──────────────────────────────┬──────────────┘
                  │                              │

   ┌──────────────────────────────┐   ┌──────────────────────────┐
   │  L4 Knowledge                │   │  L5 Model                │
   │  center.knowledge            │◄─►│  center.model            │
   │  center.asset                │   │  LLM/Embed/Flux/Video/TTS│

   └──────────────┬───────────────┘

   └─ center.qa：横切，被所有 factory 调用做一致性与设定校验
```

---

# 4. 各层职责

| 层 | 模块 | 职责 | 不负责 |
|----|------|------|--------|
| L1 Presentation | Open WebUI / Dashboard / Admin / CLI / API | 用户交互与展示 | 业务逻辑 |
| L2 Workflow | `orchestrator`（LangGraph） | 所有 skill 调度，节点可替换 | 具体业务实现 |
| L3 Business | 5 个 factory | 真正的内容生产（skill 组合） | 状态持久化（交给 Center） |
| L4 Knowledge | `center.knowledge` / `center.asset` / `center.qa` | 设定、资产、一致性的统一管理 | 直接生成内容 |
| L5 Model | `center.model` | 模型统一封装与调度（多后端） | 业务编排 |
| L6 Infrastructure | `infra` | 容器、队列、存储、监控底座 | 业务与模型 |

**关键约束**：L3 的 factory **不直接生成知识**，必须先 *查询* L4 → 再生成 → 再 *写回* L4，保证可追踪、可回滚。

---

# 5. Skill / Agent 协作模型

Skill 之间**不互相直接调用**，一律经编排器与任务队列（术语见 `12-术语与Skill规范.md`）：

```
User → orchestrator → Task Queue → Skill → Tool → Center → Return
```

所有 skill 统一具备：标准化 input/output schema、日志、Memory、Prompt、工具集、评测钩子。任何 skill 节点均可独立替换。

---

# 6. Factory 内部结构（skill 序列）

| Factory | 内部流水线（skill 序列） |
|---------|-----------|
| `factory.novel` | Outline → Plot → Chapter → Dialogue → QA → Export → **Screenplay** |
| `factory.visual` | Reference → Character Sheet → Image Gen → LoRA Train → 入库 `center.asset` |
| `factory.animation` | **Storyboard**（生产镜头表）→ Camera → Keyframe → Video → FX → Render |
| `factory.audio` | TTS → Voice Clone → Music → Foley → Subtitle |
| `factory.publishing` | Compliance Check → Format → 多平台适配 → 推送 → 回传数据至 `center.operation` |

> **Screenplay vs Storyboard 边界**（见 `43` §2.7 注释）：
> - `skill.novel.screenplay`（`factory.novel` 末）= **文学意图层**：逐章将正文拆解为含情绪/心理/对话/潜台词/构图意向的 Beat Cards，由 LLM 生成，是视觉和动画的**需求规格**。
> - `skill.anim.storyboard`（`factory.animation` 首）= **生产镜头表**：在 Screenplay 基础上，美工/AI 将 Beat 合并或拆分为具体镜头，标注资产 AssetID、精确时长与分辨率，是送进渲染管线的**执行文件**。
> - `22` 标准 #3"能输出结构化分镜"由 **screenplay 满足**（字段完整率即 Beat Card 必填字段齐全）；storyboard 是 M4 动画 MVP 才进入的后续节点。

详细 skill 拆解见 `62-AgentSkills实现规范.md` §7。

# 7. 全局数据流

```
Idea → Knowledge → Novel → Screenplay(意图) → [Asset → Storyboard(生产)] → Video → Voice → Music → Editing → [Compliance] → Publishing
```

> `Screenplay`（意图层，`factory.novel` 产出）→ 驱动 `factory.visual`/`factory.animation`；
> `Storyboard`（生产层，`factory.animation` 首节点）→ 在 Screenplay Beat 上叠加 AssetID 与渲染参数。

每一步均写入 `center.knowledge` / `center.asset` 并携带 lineage（见 `61-数据契约与Schema.md`），经 `center.qa` 校验，实现全链路可追踪、可回滚。

---

# 8. Asset Center 引用规则

`center.asset` 统一管理：Characters / Costumes / Props / Buildings / Maps / FX / Voice / Animation / Prompt / LoRA / Reference / Version。

铁律：所有 factory **只能引用 AssetID**（三段式格式见 `11` §2.2），不得脱离 ID 重新生成同一资产。`factory.visual` 是唯一有权"新建"视觉资产并写入 `center.asset` 的工厂。

# 9. 仓库结构（Monorepo）

```
backend/      frontend/     orchestrator/
factories/    centers/      infra/
configs/      database/     assets/
models/       scripts/      docs/
tests/        deploy/
```

`factories/` 与 `centers/` 下的子目录名直接对应 Canonical ID（如 `factories/novel/`、`centers/asset/`）。统一 Git Flow、统一版本、统一 CI/CD。

# 10. 开发顺序（3 人并行 + Mac 边界）

| 阶段 | 内容 | 主负责 |
|------|------|--------|
| 一 | `infra` → `center.knowledge` → `factory.novel`（本地 LLM/Mac） | 代码 + 剧情 |
| 二 | `center.asset` → `factory.visual`（Image 本地试 / LoRA 云 GPU） → **一致性验证关卡（M3.5）** | 美工 + 代码 |
| 三 | `factory.animation`（视频云 GPU） → `factory.audio`（TTS 本地 / Music） | 美工 + 代码 |
| 四 | `factory.publishing` → `center.operation` → 优化 | 全员 |

> 第二阶段末尾的一致性验证关卡，未通过不得进入第三阶段动画流水线。详见 `52-一致性技术预研方案.md`。

# 11. 技术原则

① **分层 Local First**：能力优先本地完成，**但按硬件边界分层**——
　- Local（Mac 擅长）：LLM 推理 / TTS / 编排 / 知识库 / 设定 QA。
　- Cloud-burst（CUDA 必需）：LoRA 训练 / 视频生成（Mac 的 MPS 跑不动或极慢，见 `32` §3）。
② **AI Native**：Skill/Agent 是核心而非插件。
③ **Asset Driven**：所有成果沉淀为资产。
④ **Knowledge Driven**：知识先于生成。
⑤ **Service Oriented**：每个能力都是独立服务/skill。
⑥ **Replaceable**：模型可替换——明确限定在 `center.model` 层（多后端，见 `33`）；`orchestrator` 深度绑定 LangGraph，编排框架本身不在"可随意替换"承诺范围内。
