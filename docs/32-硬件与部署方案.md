# 1. Mac Studio 配置选型

Mac Studio 采用 Apple Silicon 统一内存（Unified Memory），**内存档位直接决定可本地运行的模型上限**。

| 统一内存 | 可跑 LLM（量化后）参考 | 适用 |
|---------|----------------------|------|
| 64 GB | ≤ 34B（4-bit）流畅；70B 勉强 | 入门，写作够用 |
| 128 GB | 70B（4-bit）流畅；并发余量 | **推荐起点** |
| 192 GB+ | 70B（更高精度）/ 多模型常驻 | 多任务并行/审校+写作双模型 |

> 选型建议：以**能流畅跑 70B 级中文模型的 4-bit 量化版**为目标内存档位（≥128GB），为写作+审校双模型常驻预留空间。具体数额并入 `23` Capex。

# 2. 本地推理栈

| 方案 | 特点 | 建议用途 |
|------|------|----------|
| **Ollama** | 易用、模型管理方便、OpenAI 兼容 API | 默认接入 `center.model` 的本地后端 |
| **MLX** | Apple 原生框架，Apple Silicon 优化最佳 | 追求极致本地推理性能/微调 |
| **llama.cpp** | GGUF 生态、量化灵活 | 跨平台兜底、嵌入式调参 |

> `center.model` 对上层暴露统一 OpenAI 兼容接口，底层在 Ollama/MLX/云 API 间可切换（见 `33`）。

# 3. MPS vs CUDA 能力矩阵（关键边界）

| 任务 | Mac（MPS/MLX） | 结论 |
|------|---------------|------|
| LLM 推理（写作/审校） | ✅ 良好 | **本地** |
| Embedding / 检索 | ✅ 良好 | **本地** |
| TTS / 配音 | ✅ 良好 | **本地**（有声书主力） |
| 音乐 / 音效 | ✅ 多数可本地 | 本地为主 |
| 图像生成（SD/Flux 推理） | ⚠️ 可跑但慢 | 小批量本地试色 / 放量上云 |
| **LoRA / 扩散训练** | ❌ CUDA 生态（xformers/bitsandbytes） | **云 GPU 突发** |
| **视频扩散生成** | ❌ 多为 CUDA-only | **云 GPU 突发** |

> 这条边界是全项目的硬约束：**美工的 LoRA 训练与动画的视频生成必须规划云 GPU**，不要假设 Mac Studio 能本地完成。它也从硬件层面支持 `21` 的"有声书/动态漫优先、全动画推迟"。

# 4. 云 GPU 突发策略

| 要素 | 做法 |
|------|------|
| 触发时机 | 仅在 LoRA 训练、视频生成、批量图像时按需租用 |
| 计费 | 按小时/按量；用完即停，不包年（见 `23` §5） |
| 镜像 | 预制含 CUDA + 一致性工具链（ControlNet/IP-Adapter/ArcFace）的镜像，秒级开机 |
| 数据流 | 本地 Mac 准备 input（spec/参考图）→ 上云生成 → 产物回传 NAS 并注册 `center.asset` |
| 选型 | 优先支持按需的 GPU 云（A100/4090 级别按任务选） |

# 5. 存储与 NAS

| 数据类型 | 存放 | 说明 |
|---------|------|------|
| 模型权重 | 本地 SSD / NAS | GGUF/MLX 量化模型 |
| 资产（图/音/视频） | NAS / MinIO | 按 AssetID 组织（见 `61`） |
| 知识库 | PostgreSQL | 设定/时间线/关系（见 `42`、`61`） |
| lineage / 版本 | PostgreSQL + 对象存储 | 可回滚（见 `61` §4） |

# 6. 网络拓扑与 Docker 部署（infra）

```
Mac Studio (本地主机)
 ├─ Docker: infra(Redis/Celery/Nginx/MinIO/监控)
 ├─ Ollama / MLX (center.model 本地后端)
 ├─ PostgreSQL (center.knowledge / lineage)
 └─ orchestrator + factories(novel/audio) + centers
        │  (突发任务)
        ▼
云 GPU (按需)
 └─ factory.visual(LoRA/批量图) · factory.animation(视频)
```

- `infra` 的容器、队列、存储、监控、备份全部在 Mac Studio 本地。
- 云 GPU 仅承载视觉/视频突发 skill，产物回传本地注册。

# 8. 备份、监控与灾备预案

## 8.1 备份策略

| 数据 | 频率 | 存储位置 | 保留期 |
|------|------|---------|--------|
| PostgreSQL（知识库/lineage） | 每日增量 + 每周全量 | NAS + 异地云存储 | 90 天 |
| 资产文件（图/音/视频） | 新产物即时同步 | NAS（主）+ 云对象存储（副） | 永久 |
| 模型权重（GGUF/MLX） | 拉取后本地固化 | 本地 SSD + NAS | 永久 |
| Docker 配置/compose | 随代码提交 Git | GitHub | — |

## 8.2 监控指标

- 服务存活（orchestrator/PostgreSQL/Ollama/Redis/Celery）：健康检查每 60 秒
- 队列积压：Celery 队列长度 > 50 时告警
- 流水线成功率：对接 `22` 标准 #5、`72` §2.3
- 磁盘用量：SSD/NAS > 80% 时告警
- 稳定性目标：核心服务 **≥ 30 天无人工重启**

## 8.3 灾备预案（Mac Studio 单点故障）

> Mac Studio 承载全部 infra，是当前唯一物理节点（单点故障）。3 人小团队接受该风险，但须有明确的恢复路径。

**RTO（恢复时间目标）：≤ 4 小时**（Phase 1 接受；Phase 2 规模化后收紧到 ≤ 1 小时）
**RPO（恢复点目标）：≤ 24 小时**（最多丢失一天数据，靠每日备份保障）

| 故障场景 | 恢复步骤 | 预期时长 |
|---------|---------|---------|
| 服务崩溃（非硬件） | `docker compose restart` / `supervisorctl restart` | < 10 min |
| Mac 系统崩溃（可重启） | 重启 → docker compose up -d → 验证服务 | < 30 min |
| Mac 硬件故障（需送修） | ① 从 NAS 恢复 PostgreSQL；② 临时借用 MacBook/云主机起最小服务；③ 数据恢复后迁回 | 2–4 h |
| NAS 同时故障 | 从异地云存储恢复最近一次全量备份 | 4–8 h（可接受，极低概率） |

## 8.4 恢复演练（每季度一次）

- 从 NAS 备份恢复 PostgreSQL 到测试实例，验证数据完整性
- 确认 Docker compose 从 Git 可一键拉起全部服务
- 记录实际耗时，若超 RTO 则优化脚本

> **恢复脚本**放在 `scripts/disaster_recovery/` 目录，包含 `restore_db.sh` / `start_services.sh`，可在无文档情况下独立执行。
