# 2. 命名约定（Naming Convention）

## 2.1 模块 ID 格式

```
{type}.{name}
```

`type` 仅允许以下四类，含义不可混用：

| type | 中文 | 定义 | 关键判据 |
|------|------|------|----------|
| `infra` | 基础设施 | 容器、队列、存储等底座 | 被所有层依赖，不含业务 |
| `center` | 中心 | **持久化共享服务**：存储、管理、对外提供查询 | 被"查询/读写"，不被"运行" |
| `factory` | 工厂 | **生产流水线**：把输入按有序阶段转换为输出 | 被"运行/执行"，有明确 input→output |
| `service` | 服务 | 横切能力，被多个 factory 调用 | 跨模块、无独立产物 |

> **核心区分原则**：
> **Factory 负责"生成"，Center 负责"存储与服务"。**
> 凡是"产出新内容的流水线"→ factory；凡是"沉淀、管理、被引用的状态"→ center。
> 这条原则直接解决 Asset Center vs Asset Factory 的混乱（见 §4）。

## 2.2 实体 ID 格式

资产 / 知识实体统一为三段式：

```
{module}.{entity}.{id}
```

示例：

```
asset.character.0001        # 角色资产
asset.lora.0001-v3          # 该角色第 3 版 LoRA
knowledge.timeline.0042     # 时间线条目
knowledge.sect.0007         # 宗门设定条目
```

所有 Agent 只能**引用 ID**，不得脱离 ID 重新生成（呼应 Asset First 原则）。

## 2.3 落地约束

同一 Canonical ID 在以下位置必须**字面一致**：

- 文档标题与正文
- 代码目录（`backend/factories/novel/`、`backend/centers/asset/`）
- 数据库 schema / 表前缀（`factory_novel_*`、`center_asset_*`）
- Docker service 名（`factory-novel`、`center-asset`）
- Agent 名称（`agent.novel.outline`）

# 3. 权威模块清单（Canonical Module List）

## 3.1 分层（Layers，非模块，是架构层级）

| Layer | 名称 | 说明 |
|-------|------|------|
| L1 | Presentation | 展示层，仅交互不含业务 |
| L2 | Workflow | LangGraph 编排层，`orchestrator` 所在层 |
| L3 | Business | 业务层，所有 `factory.*` 所在层 |
| L4 | Knowledge | 知识层，`center.knowledge` / `center.asset` / `center.qa` |
| L5 | Model | 模型层，`center.model` |
| L6 | Infrastructure | 底座 `infra` |

## 3.2 模块（Modules）

| Canonical ID | 中文名 | 类型 | 所在层 | 职责 |
|--------------|--------|------|--------|------|
| `infra` | 基础设施 | infra | L6 | Docker / Redis / Celery / Nginx / 监控 / 备份 |
| `orchestrator` | 编排器 | service | L2 | LangGraph，全局任务调度 |
| `center.knowledge` | 知识中心 | center | L4 | 世界观 / 人物 / 地图 / 宗门 / 法宝 / 时间线 |
| `center.asset` | 资产中心 | center | L4 | 资产**注册与版本管理**（存储，非生成） |
| `center.model` | 模型中心 | center | L5 | LLM / Embedding / Flux / Video / TTS 统一封装 |
| `center.qa` | 质检中心 | center/service | L4（横切） | 一致性校验、设定冲突检查（原 QA Factory） |
| `center.operation` | 运营中心 | center | 横切 | 发布数据、分析、多平台同步、IP 资产运营 |
| `factory.novel` | 小说工厂 | factory | L3 | Idea→Outline→Chapter→Review |
| `factory.visual` | 视觉工厂 | factory | L3 | 立绘 / 场景 / 法宝 / LoRA **生成**（产物入 `center.asset`） |
| `factory.animation` | 动画工厂 | factory | L3 | 分镜 / 运镜 / 视频生成 / 渲染 |
| `factory.audio` | 音频工厂 | factory | L3 | TTS / 配音 / 音乐 / 音效 / 字幕 |
| `factory.publishing` | 发布工厂 | factory | L3 | 多平台适配与推送（原 Publishing Center） |

# 4. 旧名 → 新名 映射表（迁移用）

| 旧称（出现处） | Canonical ID | 处理说明 |
|----------------|--------------|----------|
| Asset Center（规划书） | `center.asset` | 保留为**存储/注册**职责 |
| Asset Factory（架构文档） | `factory.visual` | "生成"部分拆出，独立为视觉工厂 |
| Operation Center（规划书） | `center.operation` | 架构文档遗漏，**补回** |
| QA Factory（架构文档） | `center.qa` | 改为横切质检中心，非独立工厂 |
| Publishing Center（规划书） | `factory.publishing` | 其本质是推送流水线，归为 factory |
| Media Factory（顶层图） | **废弃** | 非真实模块，仅为 animation+audio 的口语化统称；停止使用 |
| Model Center / Model Layer | `center.model` | 统一为 `center.model` |

> **重点说明 1 — Asset 的拆分**：
> 原文档把"生成视觉资产"和"管理视觉资产"混为一个模块，导致 Center/Factory 命名摇摆。
> 现明确：`factory.visual` **生成** → 产物注册进 `center.asset` **存储** → 其他工厂只 **引用 AssetID**。
> 三者关系：`factory.visual` —产出→ `center.asset` —被引用→ `factory.animation`。

> **重点说明 2 — QA 的定位**：
> QA / 一致性校验是**横切能力**（小说要查设定冲突、视觉要查角色一致、动画要查帧间稳定），
> 不应作为与 novel/animation 并列的独立工厂。现定为 `center.qa`，被所有 factory 调用。
> 鉴于"一致性"是本项目核心技术风险，`center.qa` 的能力建设另见独立预研方案。

> **重点说明 3 — Media Factory 废弃**：
> 顶层架构图中的 Media Factory 在两份模块清单里都不存在，属于绘图时的临时聚合词。
> 正式架构中**不存在该模块**，动画与音频分别为 `factory.animation` / `factory.audio`。
