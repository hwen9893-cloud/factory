# 1. 文档目的

把分散在各文档的验收数字（`22` 成功标准、`52` M3.5、`21` 市场信号）变成**可复跑的评测脚本与基线**，避免"有指标却无法测量"。评测是贯穿每个 Phase、为下一个决策节点提供数字依据的持续行为。

# 2. 三类评测套件

## 2.1 市场评测套件（Phase 0，内容评价）

| 指标 | 采集源 | 判定 |
|------|--------|------|
| 完播率 | 平台后台 | 对标同品类中位 |
| 追更意愿 | 关注增量 + 评论催更 | 对标同品类中位 |
| 互动率 | 点赞/评论/转发 | 对标同品类中位 |
| 涨粉转化 | 发布周涨粉 | 对标同品类中位 |
| 评论情绪 | 评论文本情绪分类 | 正/中/负占比 |

> 输出绿/黄/红判定（`21` §1.4），喂给 Phase 决策评审。

## 2.2 内容/一致性评测套件（Phase 1–2）

| 指标 | 测试方法 | 基线 | 来源 |
|------|----------|------|------|
| 设定冲突召回 | 注入人工标注冲突集，QA 检出 | ≥ 80% | `22` #1 / `52` |
| 设定冲突密度 | 人工评审每卷冲突数 | < 2 处/卷 | `22` #2 |
| 人工修改率 | 需改动句子占比 | ≤ 20% | `22` #2 |
| 角色量化分布 | `41` 七维加权总分 | S 级 ≤ 2 主角向 | `41` §八 |
| 身份一致 | ArcFace 余弦相似度 | ≥ 0.6 / 一致率 ≥ 90% | `52` §4 |
| 跨模态命中 | 外观属性比对 spec | ≥ 85% | `52` §4 |
| 时序一致 | 10 秒片段帧间漂移 | 人工通过 ≥ 80% | `52` §4 |

## 2.3 系统评测套件（Phase 1+，工程评价）

| 指标 | 测试方法 | 基线 | 来源 |
|------|----------|------|------|
| Beat Card 字段完整率 | screenplay 必填字段（5组）齐全检查 | 100% | `22` #3 |
| 镜头意图人工可用率 | 美工评审 Beat Card 无需大改占比 | ≥ 80% | `22` #3 |
| lineage 完整率 | 产物携带 lineage 比例 | 100% | `22` #4 / `61` |
| 流水线成功率 | novel→screenplay 端到端 | ≥ 95% | `22` #5 |
| 合规通过率 | skill.qa.compliance 自动审核通过比例 | ≥ 99%（剩余 1% 进人工） | `45` §4 |
| 稳定运行 | 无人工重启天数 | ≥ 30 天 | `22` #5 |

## 2.4 经济性评测套件（Phase 1+，成本追踪）

| 指标 | 采集方式 | 目标 | 来源 |
|------|---------|------|------|
| 单章变量成本 | 记录 LLM/TTS token 用量 × 单价 | 持续下降趋势 | `23` §7 |
| 单分钟视频变量成本 | 记录云 GPU 时长 × 单价 | 持续下降趋势 | `23` §7 |
| 人工工时/章 | 三人各自记录修改时间 | ≤ 20 min/章（对应修改率 ≤ 20%） | `22` #2 |
| 变现收入/月 | 各平台后台收益汇总 | 正增长 | `23` §6 |
| 盈亏平衡月份 | 变现 ≥ Opex 的第一个月 | 尽早 | `23` §7.3 |

> 成本数据每月更新至 `23` §7.2 核算表；季度复盘时判断是否已到规模化临界点。

# 3. 测试集构建

| 测试集 | 内容 | 维护人 |
|--------|------|--------|
| 设定冲突集 | 人工注入的属性/时间线/关系冲突样本 | 剧情 + 代码 |
| 合规红线集 | 覆盖 §4 六类风险的正负样本（各 ≥ 20 条） | 代码 |
| 参考图集 | 各主角 canonical reference sheet | 美工 |
| 外观属性清单 | 每角色 K 个可比对属性（对接 `61` spec） | 美工 + 剧情 |
| 成本日志 | 每次流水线运行记录 token/GPU 用量（自动） | 代码 |

# 4. 度量脚本与基线冻结

```
W1（预研首周，见 `52` §5）：
  1. 写度量脚本：embedding 相似度 / 属性命中 / 冲突召回 / 流水线成功率
  2. 跑基线数据（2 周真实数据）
  3. 反推合理阈值，冻结为该版本验收线（避免假精确）
```

- 脚本可复跑、可 CI 集成。
- 阈值"待校准 → 冻结"后写回 `22` / `52` 对应表。

# 5. 回归检测

| 触发 | 动作 |
|------|------|
| 更换模型/量化档位（`33`） | 跑内容评测套件，对比基线，防中文质量悄退 |
| 一致性方案调整（`52`） | 跑一致性套件，检测漂移 |
| 连载推进（每卷） | 跑设定冲突 + 角色量化，检测长程漂移 |
| 合规规则变更（`45`） | 跑合规红线集，验证新规则未破坏已有逻辑 |

# 6. 与决策节点的对接

| 决策节点 | 依赖评测套件 |
|---------|-------------|
| Phase 0 形式决策 | 市场评测套件（§2.1） |
| Phase 1 验收 | 内容 + 系统套件（§2.2/§2.3） |
| M3.5 关卡 | 一致性套件（§2.2 后半） |
| 模型选型冻结 | 内容套件盲测（`33` §3） |
| 规模化决策 | 经济性套件（§2.4）：变现 ≥ Opex 且成本下降趋势稳定 |
| 合规变更后 | 合规红线测试集回归（`45` §4） |

> 一句话：**评价不是最后才做的事，它为每个决策节点提供数字依据。**

# 7. 读者反馈 → 创作闭环契约

> Market-First 的精髓不只是"上线看数据"，而是**用数据指导下一轮创作**。本节定义数据如何回流到 `factory.novel`。

## 7.1 反馈信号 → 创作调整映射

| 信号（§2.1 采集） | 判断 | 创作调整动作 |
|-----------------|------|------------|
| 完播率 < 同品类中位 | 开篇钩子弱/节奏问题 | 调整 `43` outline 爽点分布（前 3 章必有 A 级触发） |
| 追更率高但完播率低 | 中段拖沓 | `43` plot 场景卡减少 slow pacing 比例 |
| 评论高频词 = 某角色名 | 该角色有热度 | 下一卷增加该角色 Focus 章节；`41` 量化重评提权重 |
| 评论高频负面 = "人设崩了" | 角色行为与设定不符 | 复查 `center.qa` 召回率，收紧规则集（`42` §4） |
| 某形式完播明显高于另一形式 | 形式选择信号 | Phase 决策评审的 🟢 输入，调整资源分配 |
| 特定爽点类型评论热度高 | 偏好识别 | 更新 `44` §6 style_hints，该爽点频次上调 |

## 7.2 闭环操作节奏

```
每集/每章发布后（自动脚本）：
  → 平台数据导入 §2.1 指标看板
  → 评论情绪分类（正/负/关键词 top10）

每卷完成后（剧情设计，≤ 1 天）：
  → 查看信号汇总，对照 §7.1 判断调整项
  → 更新下一卷 outline 爽点策略 / 角色 Focus
  → 若调整幅度大，更新 `41` 量化权重并归档版本

每 Phase 末（全员决策）：
  → §2.4 经济性数据判断规模化时机
  → §2.1 市场数据判断是否 pivot 内容
```

> **最小形式**：剧情设计花 30 分钟看完播/追更/评论关键词 → 直接影响下一卷大纲爽点安排。不需要复杂数据分析系统。
